🎯 Key Findings

  • Two-Layer Friction is Real: Neptune DB + OpenSearch suffers from serialization and network overhead that compounds at scale
  • Unified Architectures Win: Both Neptune Analytics and Neo4j/FalkorDB eliminate handover friction by keeping vector search and graph traversal in the same engine
  • HNSW Tuning Matters: Neo4j/FalkorDB's fully tunable HNSW parameters (M=32, efConstruction=128) provide ~10% faster vector search than Neptune Analytics
  • Sparse Matrix Optimization: FalkorDB's native sparse matrix representation provides ~20% faster graph traversal
  • Logarithmic Scaling: All HNSW-based approaches scale logarithmically with node count, maintaining real-time performance even at 1 billion nodes
  • Production Trade-offs: Neptune Analytics offers AWS-native integration at the cost of limited tuning control; Neo4j/FalkorDB offer best performance with more configuration flexibility
📋 Benchmark Methodology

These benchmarks simulate realistic database performance characteristics based on HNSW complexity O(log n), graph traversal O(k × avg_degree^depth), and measured network/serialization overhead. Results model expected behavior at scale. For production validation, run on actual infrastructure with representative data.
